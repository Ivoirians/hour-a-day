I'm' continuing to work on this tiny widget. Most components can be swapped out without too much effort--specifically, the datetime selectors and the data source. I really want to use a graph like the one below, and maybe I can just use it as a static image and include a slider. But there are more data points than pixels on a screen, so I'll need a zoom tool, and that'll get complicated.

I'm reading the entire csv file as a string and splitting along new lines, then commas to pull out the individual cells. I then create an object (one of the niftier pieces of Javascript so far), which essentially functions as a hash map--I can handily store the opening/closing numbers on each date. I'm storing everything as a string because, for the most part, there are way more data points than will be necessary, so converting things into datetimes or numbers isn't necessary for the vast majority of entries. Now, given a day, I can either directly pull the correct numbers out of the object if the day is in the object, or increment the day one by one until I find a day that the S&P 500 opened on. Fun fact: the longest closure of the S&P 500 since 195 was for 7 days... starting on September 11th, 2001. Maybe a not so fun fact.

Javascript has been cooperating so far. Parsing out datetimes into the same format as the CSV file turned out to be a slight pain. The internet directed me to use module called date.format. I'm still somewhat averse to using node/npm yet, half because I've got other things to worry about right now, half because I want to do as much as I can by myself. And oh, half because of that whole left-pad mess. It might be unnecessary, but I feel better about making pages that are essentially 100% vanilla ES6 compliant.

In the end, I learned how to do some basic string/datetime parsing/manipulation and got to write a few for loops. Overall, probably a pretty good start if I had zero CS background and was learning my first language. I don't know how much I learned from this effort, but at least I made something, and that feels pretty cool. Just from a not-so-random sample, it seems like a 7% average return is not a totally unfounded statistic.

LaTeX

Knowing me, I won't infrequently want to put some mathematical symbols in my pages. LaTeX is what most math people use to display symbols, and I've gotten pretty familiar with it. In my old site/blog, I'd write up a LaTeX expression, create an image, and embed it. A quick check online informs me that most browsers now support the HTML5 standard MathML, for which converters like LaTeXML (LaTeX to MathML). Unfortunately, after much searching, it seems that Chrome specifically doesn't include MathML support. Nothing renders. The next best bet appears to be MathJax, which supports inline LaTeX, so that's pretty cool. Next time I need to do something math-y, I'll start including MathJax. But we're low on time today.
